# JaCoCo 커버리지 리포트 해석 가이드

## 📊 Total (전체) 라인 읽기 예시
```
Total: 18,078 of 30,271 (40%) | 2,271 of 3,265 (30%) | 1,804 / 2,738 | 3,590 / 6,104 | 593 / 1,092 | 62 / 211
```

---

## 🎯 각 열의 의미

### 1️⃣ **Missed Instructions / Cov.** (명령어 커버리지)
- 퍼징 중 **실행되지 않은 바이트코드 명령어 수** / **전체 명령어 수** / **커버리지 %**
- `18,078 of 30,271 (40%)` → 30,271개 명령어 중 12,193개가 실행됨 (40% 커버리지)
- **가장 정확한 커버리지 지표** (라인 단위보다 세밀함)
- 높을수록 좋음 (80% 이상 권장)

### 2️⃣ **Missed Branches / Cov.** (분기 커버리지)
- 조건문(if/else, switch 등)의 **실행되지 않은 분기 수** / **전체 분기 수** / **커버리지 %**
- `2,271 of 3,265 (30%)` → 3,265개 분기 중 1,093개 분기만 테스트됨 (30%)
- If문의 true/false 중 하나만 테스트되었다는 의미
- **높을수록 좋음** (좌측 분기와 우측 분기 모두 실행되었으면 가능)

### 3️⃣ **Missed / Cxty** (순환 복잡도)
- 좌측 **타겟되지 않은 분기 경로 수** / 우측 **전체 복잡도**
- `1,804 / 2,738` → 2,738개의 분기 경로 중 1,804개가 아직 실행 안됨
- 숫자가 높을수록 코드가 복잡함
- **퍼징 분석 목적**: 높을수록 더 깊이 있는 경로 탐색이 가능

### 4️⃣ **Missed / Lines** (라인 커버리지)
- **실행되지 않은 코드 라인 수** / **전체 라인 수**
- `3,590 / 6,104` → 6,104개 라인 중 2,514개 라인이 실행됨
- 프로그램이 실행한 소스 라인의 비율
- 가볍지만 명령어 커버리지보다는 덜 정확함

### 5️⃣ **Missed / Methods** (메서드 커버리지)
- **호출되지 않은 메서드 수** / **전체 메서드 수**
- `593 / 1,092` → 1,092개 메서드 중 499개 메서드가 호출됨
- 얼마나 많은 함수를 실제로 호출했는지 표시

### 6️⃣ **Missed / Classes** (클래스 커버리지)
- **아직 로드되지 않은 클래스** / **전체 클래스 수**
- `62 / 211` → 211개 클래스 중 149개 클래스가 로드됨
- 가장 높은 수준의 커버리지 (보통 가장 높은 % 표시)

## 🎨 시각적 표현 (녹색/빨강색 바)

- **🟢 녹색 바**: 퍼징 중 **실행된** 코드
- **🔴 빨간색 바**: 퍼징 중 **실행되지 않은** 코드
- 바의 길이가 클수록 더 많은 양, 빨간색이 길 만큼 미탐색 영역

### 예시
```
org.yaml.snakeyaml.scanner (87% 커버리지)
초록색 긴 바 + 빨간색 짧은 바 → 대부분 실행됨
```

## 📈 패키지별 분석 팁

현재 최고 커버리지 패키지:
1. **org.yaml.snakeyaml.resolver** (91% Instructions)
2. **org.yaml.snakeyaml.tokens** (90% Instructions)
3. **org.yaml.snakeyaml.scanner** (87% Instructions)
4. **org.yaml.snakeyaml.composer** (87% Instructions)

최저 커버리지 패키지:
1. **org.yaml.snakeyaml.emitter** (0% Instructions) ← 사용하지 않는 YAML 출력 부분
2. **org.yaml.snakeyaml.env** (0% Instructions) ← 환경변수 처리 부분
3. **org.yaml.snakeyaml.serializer** (1% Instructions) ← 직렬화 부분

**→ 퍼징이 주로 YAML 파싱(scanner, parser, composer) 쪽에 집중되고 있음을 의미**

---

## 🔍 취약점 분석에 활용하는 방법

1. **고커버리지 영역 + Crashed**: 취약점이 발견되었을 가능성 높음
   - 예: constructor (33% 커버리지) → YAML deserialization 취약점 가능

2. **저커버리지 영역**: 
   - 미탐색 경로가 남아있음 → 더 퍼징하면 새로운 크래시 가능

3. **분기 커버리지가 낮은 영역**:
   - 특정 조건문이 한쪽만 테스트됨 → 반대쪽에 버그 가능성

---

## 💡 퍼징 시간을 늘려야 하는 신호

- 전체 Instructions 커버리지 < 50% → 더 퍼징 필요
- 특정 패키지가 0% → 그 영역을 호출하는 입력값 탐색 필요
- Branches 커버리지 << Instructions 커버리지 → 모든 경로를 다 테스트하지 못함
